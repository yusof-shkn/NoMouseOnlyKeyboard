Based on the code and database schema, here's what **should happen** for each row action:

---

## 1. APPROVE
- PO status: `draft/pending` → `approved`
- No inventory change yet
- No journal entry yet
- Supplier balance updated via trigger (`trigger_supplier_balance_on_po_status`)

---

## 2. RECEIVE GOODS
- PO status: `approved` → `received`
- **Inventory**: `product_batches.quantity_available` incremented for each item
- **Journal Entry should fire**:
  - Debit: Inventory (asset)
  - Credit: Accounts Payable (supplier's sub-account)
- `purchase_order_items.quantity_received` updated
- `received_by` and `received_at` stamped

---

## 3. MARK AS PAID
- PO `payment_status`: `unpaid` → `paid` (or stays `unpaid` if partial)
- `paid_amount` incremented, `amount_due` decremented
- Payment record created in `payment_transactions`
- **Journal Entry should fire**:
  - Debit: Accounts Payable (supplier)
  - Credit: Cash/Bank (based on payment method)

---

## 4. REJECT
- PO status: `pending` → `rejected`
- No inventory change
- No journal entry
- Supplier balance reversed via trigger

---

## 5. CANCEL
- PO status: `draft/pending/approved` → `cancelled`
- No inventory change (goods never received)
- Supplier balance reversed via trigger

---

## 6. DELETE
- Soft delete only (`deleted_at` stamped)
- Only allowed on `draft` status
- No financial impact

---

## 7. RETURN (Purchase Return)
- Only on `received` POs
- **Inventory**: `product_batches.quantity_available` decremented
- `purchase_return_items` created
- **Journal Entry should fire**:
  - Debit: Accounts Payable (reduce what's owed)
  - Credit: Inventory (remove returned goods)
- Supplier balance adjusted

---
